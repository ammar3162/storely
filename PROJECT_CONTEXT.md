## الجداول الأساسية بقاعدة البيانات

| الجدول | الوصف |
|---|---|
| `organizations` | المؤسسات (المستأجرين) |
| `branches` | الفروع — كل مؤسسة لها فرع واحد أو أكثر |
| `profiles` | المستخدمون (مرتبطون بـ org_id) |
| `products` | المنتجات (مرتبطة بـ org_id و branch_id) |
| `purchases` | فواتير المشتريات |
| `stock_movements` | سجل حركات المخزون (وارد/صادر) |
| `suppliers` | الموردون (ميزة جديدة — يونيو 2026) |
| `supplier_order_logs` | سجل طلبات التوريد التلقائية |
| `whatsapp_logs` | سجل رسائل الواتساب المرسلة |
| `whatsapp_sessions` | حالة جلسات بوت الواتساب (تخزين دائم) |
| `notifications` | الإشعارات |
| `health_check_logs` | سجل فحوصات صحة البيانات التلقائية |

**دالة مهمة:** `my_org_id()` — دالة SQL بـ`SECURITY DEFINER` ترجع org_id للمستخدم الحالي، تُستخدم بكل RLS policies. **لا تُزال خاصية SECURITY DEFINER منها أبداً** — إزالتها تسبب recursion لا نهائي بالـ RLS (صار هذا قبل، سبب توقف الداشبورد بالكامل).

## قواعد معمارية مهمة (لا تُكسر)

1. **لا يوجد ملف `middleware.ts`.** منطق التوجيه/الحماية موجود فقط بـ `src/proxy.ts`. إضافة middleware.ts تسبب فشل البناء على Vercel (`ENOENT: .next/server/middleware.js.nft.json`) — صار هذا واتكلف ساعات تشخيص.

2. **كل استعلام منتجات يفلتر بـ `branch_id`** من `sessionStorage.getItem('s_branch_id')` — هذا التصميم **مقصود**: المستخدم يشوف بيانات الفرع المحدد فقط، ليس كل المؤسسة. أي استثناء عن هذي القاعدة (مثل لوحة "آخر الحركات") يجب أن يتسق مع باقي الصفحة.

3. **أي إضافة منتج جديد لازم تحدد `branch_id` صراحة.** سابقاً صفحة `/inventory` كانت تُدرج منتجات بدون `branch_id`، صار يسبب `null` ويُستبعد المنتج من كل الإحصائيات. تم إصلاحه (انظر السجل التاريخي).

4. **التحقق من "صنف موجود" بالمشتريات لازم يستخدم `.limit(1)` بدل `.maybeSingle()`** — لأن `.maybeSingle()` يفشل صامتاً لو وُجد أكثر من صف مطابق (وهذا يحصل لو فيه منتجات مكررة قديمة بالبيانات).

5. **صفحة الأدمن (`/storely-admin`) يجب تستخدم service role key عبر API endpoint** (`/api/admin/list-users`)، **وليس** الـ client العادي — لأن الـ client العادي يخضع لـ RLS ويُظهر فقط مؤسسة المستخدم الحالي.

6. **متغيرات البيئة:** الأسماء الصحيحة المستخدمة بالكود هي `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`. **تنبيه:** كان فيه نسخ قديمة بـ Vercel باسم مكتوب خطأ (`SUPABASE` بدل `SUPABASE`، حرف ناقص) — هذي غير مستخدَمة بالكود ويفضّل حذفها لتفادي اللخبطة مستقبلاً.

## أنظمة المراقبة والحماية (مبنية يونيو 2026)

- **Endpoint:** `GET /api/health-check`
- **يفحص:** منتجات بدون فرع، مؤسسات بدون فرع فعّال، منتجات مكررة الاسم، مستخدمين معلّقين +7 أيام، اشتراكات منتهية لكن نشطة

- **يحتاج جدولة يومية بـ cron-job.org** (method: POST) — تأكد أنها مفعّلة
- **صفحة المراجعة:** `/storely-admin/backups` — عرض وتحميل الملفات فقط (لا يوجد زر استعادة تلقائية، بقرار: الاستعادة تتم يدوياً عبر SQL بعد مراجعة بشرية، لتفادي حذف بيانات عملاء بالخطأ)

## سجل تاريخي — أخطاء حرجة تم اكتشافها وإصلاحها

> هذا القسم مهم لفهم "ليش الكود مكتوب هيك" — لا تعيد نفس الأخطاء.

1. **RLS كان معطّل بالكامل على `profiles` و`branches`** (ثغرة أمنية خطيرة) — اكتُشف عبر تنبيه أمني من Supabase. تم تفعيله، وصار recursion بسبب دالة `my_org_id()` بدون `SECURITY DEFINER` — تم إصلاحه.

2. **صفحة الأدمن كانت تُظهر مستخدم واحد فقط** بدل كل المستخدمين — سبب: استخدام client عادي يخضع لـ RLS. تم إصلاحه بـ API endpoint بـ service role key.

3. **باغ تكرار المنتجات بالمشتريات** (الأخطر والأكثر تعقيداً): اختيار "صنف موجود" كان يُنشئ منتج جديد بدل تحديث الموجود. السبب المزدوج: (أ) فلتر `branch_id` خاطئ بالتحقق بينما القائمة المعروضة بدون فلتر، (ب) `.maybeSingle()` يفشل صامتاً مع وجود تكرارات سابقة. أدى هذا لتراكم 13 مجموعة منتجات مكررة عبر مؤسستين — تم تنظيفها بدمج الكميات ونقل سجلات الحركة قبل الحذف.

4. **منتجات بـ `branch_id = NULL`** عبر 3 مؤسسات (36 منتج إجمالي) بسبب صفحة `/inventory` التي كانت لا تحدد branch_id عند الإضافة. مؤسسة EVORA لم يكن لها أي فرع مسجّل إطلاقاً. تم الإصلاح والتنظيف.

5. **متغيرات بيئة فاضية على Vercel** (`NEXT_PUBLIC_SUPABASE_URL` بدون قيمة فعلية رغم وجود الاسم) — اكتُشف عند محاولة تشغيل سكريبت اختبار محلي. الموقع المنشور لم يتأثر لأن القيم محقونة وقت البناء، لكن أي بناء جديد كان سيفشل. تم الإصلاح.

6. **تسريب مفتاح `SUPABASE_SERVICE_ROLE_KEY` بالمحادثة** عن طريق الخطأ (نسخ القيمة الكاملة بدل الاسم) — تم توليد مفتاح جديد فوراً (rotate) من Supabase Dashboard وتحديثه بكل مكان.

## مهام مؤجلة / غير مكتملة

- **صفحة هبوط تسويقية عامة** (تشبه loyapro.com) لعرض Storely للعملاء المحتملين — توقفت لأن قائمة العملاء الحاليين تحتوي حسابات تجريبية (تجربتي، تيست، ammar.16) لم يُحدَّد بعد أيها عملاء حقيقيون يُعرضون
- **سياسة الخصوصية / شروط الاستخدام / صفحة من نحن** — لم تُكتب بعد
- **تسجيل الشركة رسمياً بالسعودية** (سجل تجاري، ZATCA) — نقاش عام فقط، لا قرار
- **بوابة دفع إلكترونية** (Moyasar/PayTabs/Tap) — أولوية تالية مهمة، لم يبدأ العمل عليها
- **تحقق من عدم السماح بصرف كمية أكبر من المتاح** بالواجهة — الاختبار التلقائي كشف أن النظام يحسب قيمة سالبة رياضياً لكن لا يوجد منع فعلي بالواجهة

## أسلوب العمل المتّبع بالمحادثات

- المستخدم (عمار) ينفّذ كل أوامر Bash/SQL بنفسه على جهازه (Mac, zsh) أو بـ Supabase SQL Editor — Claude لا ينفّذ مباشرة على بيئة المستخدم.
- نمط النشر القياسي: `npm run build 2>&1 | tail -N` للتأكد محلياً، ثم `git add -A && git commit -m "..." && git push origin main && vercel --prod --force`
- عند الشك بمشاكل cache بالبناء: `rm -rf .next && npm run build`
- عند تعديل ملفات: تفضيل `python3` heredoc scripts للاستبدال الدقيق بدل sed المعقد، أو `cat > file << 'EOF'` للملفات الجديدة كاملة
